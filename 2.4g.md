# 2.4 GHz control protocol

The over-air command protocol for Neewer Infinity fixtures in 2.4 GHz mode. This is the
fixture-to-fixture path from [hardware.md](hardware.md): set fixtures to the same 2.4 GHz channel
and they follow each other. The frames documented here are what travels between them. For the
Bluetooth wire protocol, see [protocol.md](protocol.md); the two carry the same command set with a
different header.

## Link

The 2.4 GHz radio is the Si24R1 (an nRF24L01+-class part). Every fixture on a channel shares one
link:

- Modulation: GFSK at 250 kbps.
- Payload: 32 bytes, fixed length.
- CRC: 16-bit.
- Acknowledgement: none. Frames are one-way and are not acked or retried at the link layer.
- Carrier: a single fixed RF channel for every on-device channel. The on-device channel selects an
  address, not a carrier frequency (see [Addressing](#addressing)).

## Frame format

```
[0]=0x77  [1]=channel  [2]=len  [3]=sub  [4 .. 4+len-1]=payload  [4+len]=checksum
```

- `channel` is the on-device 2.4 GHz channel, counted from 0.
- `len` counts payload bytes only. It excludes the `77 ch len sub` header and the checksum.
- `checksum` is the sum of every byte from index 0 through the last payload byte, taken `& 0xFF`.
- The remaining bytes of the 32-byte payload are padding and are ignored.

The marker byte is `0x77`. The Bluetooth protocol uses `0x78` for the same commands.

## Addressing

The on-device channel does two things at once: it sets the radio pipe address, and it is written
into frame byte `[1]`.

- **Pipe address** (3 bytes): `channel+1` repeated three times. Channel 0 is `01 01 01`, channel 1
  is `02 02 02`, and so on.
- **Frame byte [1]**: the channel number itself.

A fixture only receives on its own channel's address, so a channel is a broadcast group: every
fixture set to that channel applies the frame at the same time. Separate groups live on separate
channels. There is no per-fixture unicast in this protocol; the channel is the only addressing axis.

## Commands

The sub-opcode at `[3]` selects the command. Colour and white values use the same ranges as the
Bluetooth protocol.

| sub | command | payload |
|---|---|---|
| `0x81` | power | `state` (`0x01` on, `0x02` off) |
| `0x86` | HSI | `hueLo, hueHi, sat, bri` (hue little-endian 0-360; sat, bri 0-100) |
| `0x87` | CCT | `bri, cct, gm` (bri, gm 0-100, gm 50 is neutral; cct is Kelvin / 100, so 25-100 covers 2500-10000 K) |
| `0xA8` | RGBCW | `bri, R, G, B, C, W` (bri 0-100; the six channels 0-255) |
| `0x8B` | effect | `effectId, params...` (see [Effects](#effects)) |
| `0xB2` | pixel | pixel / segment effect data |

### Examples

```
power on        77 00 01 81 01 FA
HSI blue        77 00 04 86 F0 00 64 64 B9      hue 240, sat 100, bri 100
CCT 5600 K      77 00 03 87 64 38 32 CF         bri 100, cct 0x38 = 5600 K, gm 50
RGBCW magenta   77 00 06 A8 64 80 00 80 00 00 89  bri 100, R 128, B 128
```

## Effects

An effect frame (`0x8B`) carries an effect id as its first payload byte, then the parameters for
that effect. Rates and colour values reuse the ranges above; `ember` and `mode` are effect-specific
scalars.

| id | effect | parameters after id |
|---|---|---|
| 1 | flash | `bri, cct, rate` |
| 2 | paparazzi | `bri, cct, gm, rate` |
| 3 | defective bulb | `bri, cct, gm, rate` |
| 4 | explosion | `bri, cct, gm, rate, ember` |
| 5 | welding | `briMin, briMax, cct, gm, rate` |
| 6 | CCT flash | `bri, cct, gm, rate` |
| 7 | hue flash | `bri, hueLo, hueHi, sat, rate` |
| 8 | CCT pulse | `bri, cct, gm, rate` |
| 9 | hue pulse | `bri, hueLo, hueHi, sat, rate` |
| 10 | police car | `bri, colour, rate` |
| 11 | candle | `briMin, briMax, cct, gm, rate, ember` |
| 12 | hue loop | `bri, hueMinLo, hueMinHi, hueMaxLo, hueMaxHi, rate` |
| 13 | CCT loop | `bri, cctMin, cctMax, rate` |
| 14 | intensity loop | `flag, briMin, briMax, hueLo, hueHi, cct, rate` |
| 15 | screen | `briMin, briMax, cct, gm, rate` |
| 16 | fireworks | `bri, mode, rate, ember` |
| 17 | party | `bri, mode, rate` |
| 18 | music | `bri` |

## Scope

The frame format, addressing, and command set here are documented for the TL60, TL90C, and TL120C.
The pixel payload (`0xB2`) uses the per-cell colour encoding from
[pixel-and-streamer.md](pixel-and-streamer.md).
